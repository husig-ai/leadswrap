01

Product overview

LeadNest is an AI-first CRM for real estate agents and brokerages. It replaces manual follow-up workflows with automated, personalised outreach across SMS, WhatsApp, email, and voice — anchored by a proprietary lead scoring system (LPMAMA) and a multi-source enrichment pipeline. V2 adds team support, billing, iOS, a web app, two-way communications, and Google Calendar integration.

Core problem
  • Real estate conversion drops by over 400% after the first 5 minutes of lead silence
  • Most agents manage leads in spreadsheets or generic CRMs not built for real estate
  • No existing tool combines LPMAMA scoring, AI drafting, and multi-channel comms in one app
V2 key additions
  • Multi-tenant architecture with team roles and seat-based billing
  • iOS release alongside existing Android (Apple Sign-In, APNs, App Store)
  • Full two-way SMS + WhatsApp + email with AI reply drafting on every inbound
  • Google Calendar two-way sync for showings
  • Jinja/HTMX web app (8 pages) served from FastAPI — no separate frontend server
  • Speed-to-lead auto-responder: first SMS within 60 seconds, zero agent action
  • Call intelligence: Whisper transcription + GPT-4o sentiment and action items
What V1 ships today
  • JWT auth · lead creation via form and voice
  • LPMAMA scoring with per-dimension breakdown
  • Property management and AI property matching (Gaussian decay)
  • AI-suggested first SMS with approve / edit / discard
  • Scheduled messages (Celery + Redis)
  • Push notifications via FCM (Android)
  • Lead audit trail with status transitions
  • Analytics: summary, pipeline, trend, geography, pricing overlap
  • Flutter app live on Google Play Store

V1 foundation (live backend)

The following capabilities exist in the current codebase and are confirmed working. V2 extends these — it does not rebuild them.

Data model — what is stored today

Lead entity (leads_table)

Each lead carries all LPMAMA dimensions directly on the row: preferred location, property type preference, bedrooms required, location flexibility (fixed / flexible / very_flexible), budget range with certainty rating, motivation type (relocation / first_home / upsizing / downsizing / investment / browsing), motivation urgency (urgent / planned / casual), existing agent status, mortgage status (cash_buyer / pre_approved / pre_qualified / in_process / not_started), preferred contact method, and appointment state. Status transitions follow: NEW → CONTACTED → ENGAGED → QUALIFIED → SHOWING → NEGOTIATING → LOST → CONVERTED → ARCHIVED.

Supporting tables

agents_table — profile, hashed password, phone, properties list.

messages_table — Twilio message_sid as PK, sender (agent / lead enum), status (queue / sent / delivered / received / failed), lead_id, agent_id.

reminders_table — scheduled action with Celery task_id, remind_at timestamp, action type (MESSAGE / REMINDER).

lead_scores_table — per-dimension scores, hotness tier, scoring version. Separate from hotness_score on the lead for historical tracking.

scoring_weights_table — configurable dimension weights. Loaded from DB at runtime, not hardcoded.

lead_audit_table — immutable transition log with from_status, to_status, trigger type (automatic / manual), reason, and timestamp.

Lead intake — all sources

A lead can enter LeadNest through seven distinct paths. All paths converge on the same POST /api/v1/leads endpoint, triggering an identical downstream pipeline: audit log creation, automatic LPMAMA scoring, FCM notification to the agent, and (if enabled) the speed-to-lead auto-responder.

Convergence guarantee: regardless of how a lead arrives — manual form, voice recording, inbound WhatsApp, MLS webhook, or Zapier — the same scoring, enrichment, and notification pipeline runs within seconds. No lead enters the system unscored or unnoticed.

LPMAMA scoring engine

Every lead is scored immediately on creation using the LPMAMA framework — the real estate qualification methodology taught in agent training programs worldwide. Scores are stored per-dimension in lead_scores_table and summarised as a hotness_score (0–100) on the lead record. Dimension weights are configurable per workspace and stored in scoring_weights_table.

Configurable weights: dimension weights are loaded from the database at runtime, not hardcoded. A workspace admin can adjust the weight of each dimension without a code deployment — e.g. increase the mortgage dimension for markets where pre-approval is critical.
Dimension Default pts What is scored Scoring logic
L — Location 15 preferred_location_id, location_flexibility, property_type_preference_id, bedrooms_required Location set: 40% of pts. Flexibility: fixed=33%, flexible=20%, very_flexible=7%. Property type specified: 20%. Bedrooms specified: 7%.
P — Price 20 min_budget, max_budget, budget_certainty Budget range spread: ≤20% = full pts, ≤50% = 60%, >50% = 30%. Max-only: 40%. Certainty: defined=100%, approximate=60%, unsure=20%.
M — Motivation 15 motivation_type, motivation_urgency Type: relocation/first_home = full pts; upsizing/downsizing = 75%; investment = 62%; browsing = 12%. Urgency: urgent = full; planned = 57%; casual = 14%.
A — Agent 10 has_existing_agent, existing_agent_name No existing agent = 100%. Not disclosed = 50%. Has agent = 20% (still worth nurturing).
M — Mortgage 20 mortgage_status Cash buyer = 100%. Pre-approved = 90%. Pre-qualified = 60%. In process = 35%. Not started = 15%.
A — Appointment 15 appointment_scheduled_at, appointment_requested, preferred_contact_method Booked appointment = 67% of pts. Requested but not booked = 45%. Contact method set = remaining 33%.
Completeness 5 Overall form fill rate Bonus points for leads where all optional fields are populated. Incentivises thorough qualification.

Hotness tiers

🔥 Hot (≥75)

Call within 1 hour. Lead is qualified on all key dimensions. Ready to transact. Agent gets high-urgency notification.

🌤 Warm (50–74)

Follow up within 24 hours. Strong signals on some dimensions. May need more qualification to progress.

❄️ Cool (25–49)

Weekly drip + monthly check-in. Early stage or incomplete qualification. Nurture over time.

🧊 Cold (0–24)

Long-term drip only. Very limited qualification data or low intent signals. Automated sequence only.

Score history: every score calculation is saved to lead_scores_table with a scoring_version tag. If weights are updated, historical scores remain untouched. Rescoring produces a new row, allowing agents to see how a lead's score has evolved over time.

Communication layer

All agent-to-lead communication flows through a single unified timeline per lead. SMS, WhatsApp, email, and call notes appear chronologically regardless of channel. Every inbound message triggers an AI reply draft. The agent approves, edits, or discards — this is the core HITL (human-in-the-loop) pattern carried over from V1.

Unified AI prompt manager

Before every AI generation call, the service assembles full lead context: enrichment data, intent history, last 5 interactions across all channels, current LPMAMA score with per-dimension breakdown, and top matched properties. This context is injected into a channel-specific prompt template stored in the database with a version ID. Output format varies by channel:

SMS / WhatsApp output

Under 160 characters. Conversational, first-person. References lead's name and property interest. Ends with a clear question to invite response.

Email output

Includes subject line (personalised with lead name or property address) and structured body. Pulls Clearbit company data, matched property details, and agent's email signature.

Call note output

Concise 2-sentence summary of what was discussed and lead's current position. Action items extracted as a checklist. Sentiment tag applied to the lead record.

Speed-to-lead auto-responder

First message within 60 seconds of lead creation. No agent action required. Proven to dramatically improve lead conversion — real estate conversion drops by over 400% after the first 5 minutes of silence.

On lead creation, an SQS message is pushed to the auto-responder queue. A Lambda worker picks it up and fires the first message via the same channel the lead used to enquire (WhatsApp if WA, SMS otherwise). GPT-4o personalises using lead name and property of interest. After the first message, auto-responder hands off to HITL — subsequent replies require agent approval. Workspace owners can disable auto-responder globally or per agent.

Scheduled messages (existing V1 feature)

Agents can schedule future SMS messages using POST /api/v1/reminder/schedule-message/{lead_id}. The Celery task (broker: Redis) executes at the scheduled time, sends via Twilio, and pushes a confirmation notification: "Scheduled Message Sent — [truncated content]". Tasks are cancellable via PATCH /api/v1/reminder/cancel/{id} before execution.

Property matching

LeadNest uses Gaussian decay scoring to match leads against the agent's property inventory. This produces a continuous match score (0–1) rather than a binary pass/fail, allowing subtle price proximity to score higher than a hard cutoff would. V2 extends this with intent-based curation from conversation history.

Gaussian decay algorithm (V1 — live)

For each property in the agent's inventory, two decay calculations are run against the lead's budget:

Max price decay

Origin: property.max_price. Price offset: $1,000 (deadlock zone). Scale: $3,000. Decay: 0.5 at scale. Lead's max_budget is the input value. Scores near 1.0 if budgets align, falls off smoothly as the gap widens.

Min price decay

Origin: property.min_price. Price offset: $200. Scale: $1,000. Decay: 0.5. Lead's min_budget is the input. Tighter tolerance — a lead with a low minimum near the property floor scores well.

Combined score

Final score = (2 × max_score + min_score) / 3. Max price alignment weighted 2:1 over min. Scores ≥ 0.8 → "Matched". Scores 0.5–0.79 → "Related Match". Below 0.5 → excluded from results.

Intent-based curation (V2 addition)

V2 overlays conversation context onto the property match. GPT-4o reads all interactions (SMS, WhatsApp, email, call notes) and extracts preference signals: must-have features ("big backyard", "good schools"), deal-breakers, urgency level, neighbourhood preferences. These signals are stored as structured tags on the lead record and used to re-rank Gaussian decay results. When an agent sends an AI-drafted email or SMS, top matched properties are automatically inserted into the draft for the agent to confirm.

Match score shown to agent: each suggested property displays a match score (0–100) and a one-line explanation of why it matched — e.g. "Matches: large backyard, under $500k, Westside location." This makes the agent look research-driven without manual effort.

Analytics & audit trail

All analytics endpoints are Redis-cached with configurable TTL. The audit trail is an immutable event log — no entry is ever deleted or modified. Analytics data is scoped to the authenticated agent (solo) or the full workspace (admin).

Analytics endpoints (live in V1)

Summary
  • Total leads and active leads (non-LOST/CONVERTED/ARCHIVED)
  • Conversion rate (CONVERTED / total)
  • Hotness distribution: 80–100, 60–80, 30–60, 0–30 buckets
  • Total estimated commission across all leads
Pipeline
  • Lead count per status (all 9 statuses)
  • Percentage of total per status
  • Returns zero-counts for empty statuses (not omitted)
Trend (from audit log)
  • Monthly status transition counts grouped by month/year
  • Percentage of total transitions per period
  • Derived from lead_audit_table — reflects actual history
Geography

  • Ordered by interest_count descending
  • Percentage of total per region
Pricing overlap
  • Property price distribution (bucketed)
  • Lead budget distribution (bucketed)
  • Overlap count and percentage (budget-meets-inventory)
  • Shows if inventory pricing matches lead demand
V2 additions
  • Communication response rates (SMS / WA / email / call)
  • Average call duration by agent
  • AI draft approval rate (approved vs edited vs discarded)
  • Enrichment coverage % (what % of leads have each source)

Lead audit trail

Every status transition is automatically logged by SQLAlchemy event listeners. after_insert creates the initial NEW entry. after_update on status field change creates a transition entry with old and new values. Each entry records: lead_id, agent_id, from_status, to_status, trigger_type (automatic / manual), reason, and timestamp. The full history is queryable via GET /api/v1/logs/lead-audit-logs?lead_id={id}.

Multi-tenancy & billing

Every database table includes a tenant_id column. All FastAPI endpoints extract tenant_id from the JWT and apply it as a WHERE clause. No query returns data across tenant boundaries. This is enforced at the service layer, not just the endpoint layer — even internal service calls respect tenant scope.

Tenant isolation guarantee

A multi-tenant isolation audit runs in Week 11–12. Every API endpoint is tested with two live tenant accounts — automated assertions confirm no endpoint returns data across boundaries under any condition. Test cases: cross-tenant lead access, cross-tenant property listing access, cross-tenant analytics, cross-tenant notifications.

Stripe webhook event handling

subscription.created

Feature flags updated for tenant. Trial start recorded. Seat limit set per plan.

invoice.paid

Subscription status confirmed active. Renewal date updated. Any grace-period flags cleared.

payment_failed

Admin notified (email + push). Grace period flag set. Gating activates after 7 days.

subscription.deleted

Tenant downgraded to Starter feature set. Data preserved. Seat limit reduced to 1.

Auth & identity

V1 (live)

Email + password signup and login. JWT issued with email as subject, 48-hour expiry. Password hashed with bcrypt. Logout is client-side token discard (server-side blacklisting is a TODO in the codebase, noted for V2).

V2 additions

+
Google OAuth 2.0
POST /api/v1/auth/google — backend verifies Google ID token using Google's JWKS. First login creates account (email pre-filled). Return login matches by email. Works on web (next-auth or HTMX redirect) and mobile (google_sign_in Flutter package). JWT issued after verification.
Mobile + web
+
Apple Sign-In (required for iOS App Store)
POST /api/v1/auth/apple — Apple identity token verified using Apple's public JWKS endpoint. Email returned from Apple on first login only — must be stored immediately. Subsequent logins identified by Apple user sub. Mobile uses sign_in_with_apple Flutter package. Works on device and simulator (different token flows). Required by App Store Review Guideline 4.8 for apps offering any third-party login.
App Store requirementFirst-login email only
+
Forgot / reset / change password
POST /auth/forgot-password — sends time-limited reset link to email (SendGrid). POST /auth/reset-password — validates token, updates hashed password, invalidates token. PUT /auth/change-password — authenticated endpoint, requires current password, updates to new. All three tested end-to-end.
Standard flows
+
Server-side token blacklisting
Logout stores the JWT JTI in Redis with TTL = remaining token lifetime. Middleware checks blacklist on every request. Enables true server-side logout (currently client-side only in V1 — noted as TODO in auth.py).
Redis blacklist

Infrastructure

V1 (current)

Backend

FastAPI (Python 3.12). Cloudflare Workers (migrating). SQLite for development, Cloudflare D1 for production (migrating to PostgreSQL). Redis for Celery broker and analytics cache.

Storage

Cloudflare R2 (S3-compatible) for property images and audio files. Presigned URL generation for direct client uploads.

Async tasks

Celery with Redis broker for scheduled messages and reminders. Celery task retry: max 3 attempts, 60-second delay.

V2 infrastructure

EC2 (single instance)

FastAPI on port 8000 behind Nginx (443/SSL). PostgreSQL on port 5432 (localhost only — not exposed). t3.medium (2 vCPU, 4 GB RAM). Nginx handles SSL via Let's Encrypt. Supervisor or systemd keeps FastAPI alive. Nightly pg_dump to S3. Daily EBS snapshot via AWS Backup.

Database

PostgreSQL on the same EC2 instance. All LPMAMA lead data, messages, scores, audit logs, reminders, analytics cache metadata, consent logs. Migration from Cloudflare D1 via one-time script: assigns tenant_id to all existing V1 records (idempotent, safe to re-run).

AWS SQS

Decouples lead creation from enrichment and auto-responder. Two queues: enrichment-queue and auto-responder-queue. Lambda workers consume each queue. Main API returns 201 instantly regardless of downstream processing speed.

Web app

Jinja2 templates served from FastAPI. HTMX for dynamic interactions (Kanban updates, live timeline, form submissions). Tailwind for styling. No separate Node.js server, no Vercel. Single EC2 deployment covers both API and web UI.

Mobile

Flutter — iOS and Android from single codebase. App Store (new) and Google Play Store (existing, transferring to Husig company account). Firebase FCM for push notifications. APNs certificate configured in Firebase console for iOS.

When to scale

Split DB to RDS when PostgreSQL consistently uses >60% CPU/RAM under load. Add read replica when analytics queries slow the main API. Both are one-afternoon operations — no app code changes required. Target: post-100-tenant scale.
